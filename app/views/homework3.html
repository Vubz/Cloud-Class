    const AWS = require('/usr/local/lib/node_modules/aws-sdk');
const s3 = new AWS.S3();

var params = {
 Bucket: "cc414-images",
 Key: "scientist.png"
};
s3.headObject(params, function(err, data)
{
  if (err) console.log(err, err.stack); // an error occurred
  else
  {
    if (data.Metadata.move)
    {
      move=data.Metadata.move;
      if (move=='true'){
        var params =
        {
         Bucket:'cc414-images/23042',
         CopySource: 'cc414-images/scientist.png',
         Key: 'scientist.png',
         ACL: 'public-read',
        };
        s3.copyObject(params, function(err, data)
        {
          if (err)
              console.log(err, err); // an error occurred
          else {
              console.log(data); // successful response
          }
        });
      }
      else {
        console.log("Move is false");
      }
    }
    else {
      console.log("No Metadata move");
    }
  }          // successful response
});

4. Richardson Maturity Model

Richardson Maturity Model

steps toward the glory of REST

Este escrito es una descripción de los niveles y pasos a seguir para tener un modelo REST exitoso y llegar a su máximo exponente “Glory of REST”.

Empezando con el nivel 0 (Swamp of POX) y siguiendo el ejemplo de Martin Flower para hacer una cita médica, se tendría un nivel muy básico de interacción, el cliente pide la información de citas disponibles, el proveedor manda una respuesta con los espacios disponibles, luego el cliente manda otra respuesta con la cita deseada y el proveedor responde con confirmación o error.

El nivel 1 (Resources) maneja los recursos de manera individual, la respuesta del proveedor con los espacios disponibles poseen un id único para accederse y la confirmación del proveedor es similar a la anterior pero permite al usuario acceder a su cita de manera única a través de un link con el id, Ejemplo: id=1234, http://example/citas/1234

En el nivel 2 (HTPP Verbs) se utilizan los verbos de https que caracterizan al REST, los más comunes POST, GET por ejemplo indican el tipo de cambio o pedido que se va a hacer al proveedor, eliminando variación innecesaria. También utiliza respuestas únicas determinadas por el proveedor, los códigos de respuesta indican el tipo de cambio que se hizo agregando profundidad. Los códigos empezando con 2XX indican un tipo de transacción exitosa, mientras que los códigos de error son únicos y dan una pista a la raíz del problema.

En el nivel 3 (Hypermedia Controls) se agregan URI’s a las respuestas primero como parte de estandarización y segundo para el momento de tener la transacción completada y la cita médica hecha, proveen control sobre los tipos de transacciones que se pueden hacer lo que permite actualizar una parte de la transacción sin necesidad de eliminar la transacción del cliente para logarlo.

5. Por que Cloud Computing es importante

Cloud Computing es importante debido al potencial que tiene, desde el punto de vista del programador le proporciona acceso a equipo y poder de procesamiento con valor de millones de dólares con solo una cuota al mes. Desde el punto económico tiene una similitud a la del internet, entre más personas lo usen, la red se vuelve más barata y más accesible, Cloud Computing nos pone a todos los mismos recursos para competir y trabajar para resolver problemas, crear ideas y generar negocios.
